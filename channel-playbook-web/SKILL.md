# Channel Playbook -- Web

Cargado por los Agentes B, C y D cuando web es uno de los canales activos.

La web no es un folleto digital. Es el lugar donde el cliente toma decisiones. Todo lo que aparece en pantalla debe reducir friccion, no anadirla.

Principio Maestro: **"Claridad es conversion."** Un precio claro convierte mas que un precio maquillado. Una condicion explicada convierte mas que una condicion escondida. Una pagina con un foco convierte mas que una pagina con cinco ofertas compitiendo. Este principio tiene precedencia sobre cualquier peticion de anadir mas contenido, mas banners o mas CTAs a una pagina.

El modelo que este documento reemplaza es: "Cuanto mas contenido pongamos, mas posibilidades de que el usuario encuentre algo que le interese." El modelo que este documento instaura es: "Menos saturacion. Mas claridad. Mas confianza. Mas contratacion."

---

## 1. Funcion del canal

La web cumple cinco funciones dentro del ecosistema Movistar:

1. **Cerrar decision.** La web es destino final de trafico de otros canales (email, display, Meta, tienda). Si el cliente llega, ya ha sido impactado. La web debe cerrar, no repetir el pitch.
2. **Explicar con profundidad.** Lo que no cabe en un email o un banner se desarrolla aqui. Comparativas, configuradores, detalle de producto.
3. **Construir confianza.** Claridad en precios, condiciones y beneficios. Sin letra pequena escondida. Sin asteriscos sin resolver.
4. **Capturar intencion.** Trafico organico de busqueda. El usuario viene con una pregunta; la web debe responderla.
5. **Facilitar autogestion.** Area privada, consulta de facturas, gestion de servicios. La web resuelve sin necesidad de llamar.

---

## 2. Principios clave

1. **El precio debe vender, no confundir.** Si el precio requiere tres asteriscos y letra pequena para entenderse, no esta listo para web. Simplificar antes de publicar.
2. **Primer momento de valor en tres segundos.** El usuario debe entender que gana en los primeros tres segundos de scroll. Si el hero no comunica valor claro, la pagina falla.
3. **Ventaja Personal es el centro.** En area privada y en comunicacion a clientes logados, Ventaja Personal debe sentirse como reconocimiento personalizado, no como banner generico.
4. **Un CTA por pantalla visible.** Si el usuario ve dos botones compitiendo en el mismo viewport, hay un problema de jerarquia.
5. **Cada pagina tiene una mision.** Landing de campana = conversion. Pagina de producto = decision. Area privada = gestion. No mezclar misiones.
6. **Claridad es conversion.** Un precio claro convierte mas que un precio maquillado. Una condicion explicada convierte mas que una condicion escondida.
7. **Movil primero.** La mayoria del trafico llega por movil. Si la experiencia mobile no funciona, la pagina no funciona.

## 3. Estructura de landing de campana

| Bloque | Funcion | Regla |
|---|---|---|
| Hero | Propuesta de valor en una frase + visual | Un mensaje, un CTA. Sin slider. Sin multiples ofertas. |
| Beneficio principal | Explicar por que importa | Lenguaje de beneficio humano, no de spec tecnica. |
| Prueba / RTB | Razon para creer | Dato concreto, ventaja tangible, o prueba social si aplica. |
| Detalle de oferta | Precio, condiciones, lo que incluye | Precio claro, sin asteriscos ambiguos. Condiciones visibles. |
| CTA principal | Dirigir a contratacion o siguiente paso | Boton visible, texto de accion ("Contratar", "Ver mi oferta"). |
| FAQ o resolver dudas | Anticipar objeciones comunes | Preguntas reales del cliente, no marketing disfrazado. |
| Footer | Cumplimiento y navegacion | Disclaimers necesarios + navegacion a otras secciones. |

---

## 4. Tipos de pagina y su mision

| Tipo | Mision | Que NO hacer |
|---|---|---|
| Landing de campana | Convertir trafico de pago o CRM | No llenar de productos secundarios. Un foco. |
| Pagina de producto | Ayudar a decidir con informacion completa | No esconder el precio. No abrumar con specs sin contexto. |
| Area privada | Gestionar y sentirse reconocido | No convertirla en canal de venta agresiva. |
| Landing de fidelizacion | Activar beneficios de cliente | No mezclar con captacion. Tono de reconocimiento. |
| Pagina de soporte | Resolver dudas rapidamente | No redirigir a canales cuando la web puede resolver. |

---

## 5. Coordinacion con otros canales

La web casi nunca es punto de entrada. Es destino.

- **Desde email/CRM:** El email activa. La web resuelve. El mensaje de la landing debe continuar el del email, no repetirlo.
- **Desde display/Meta:** El banner genera interes. La landing profundiza y cierra. No replicar el banner como hero.
- **Desde tienda:** Si la tienda dirige a web para ampliar informacion, la pagina debe estar preparada para ese flujo (QR, URL corta).
- **Desde search:** El usuario viene con intencion. La landing debe responder a esa intencion directamente.

Regla de coherencia: si el CTA del email dice "Descubre tu ventaja", la landing debe abrir con esa ventaja visible. No con un hero generico.

## 6. Riesgos y anti-patrones

| Anti-patron | Por que falla |
|---|---|
| Saturacion de ofertas | Multiples banners y CTAs compiten. El usuario no sabe donde mirar. |
| Precio con asteriscos sin resolver | Genera desconfianza. El cliente asume que hay trampa. |
| Hero generico | Imagen de stock sin relacion con la oferta. Pierde los tres primeros segundos. |
| Scroll infinito sin jerarquia | Informacion desordenada. El usuario abandona sin encontrar lo que busca. |
| Ventaja Personal como banner generico | Si no se siente personal, no funciona. Pierde credibilidad del mecanismo. |
| Mobile como adaptacion de desktop | El flujo mobile tiene sus propias reglas. No es "lo mismo pero mas estrecho". |
| CTA ambiguo | "Saber mas" no es un CTA. "Contratar ahora" si lo es. |

## 7. KPIs estandar

| Metrica | Para que sirve |
|---|---|
| Tasa de conversion de landing | Mide si la pagina cumple su mision |
| Scroll depth | Indica si el contenido engancha |
| Click a CTA principal | Mide efectividad de la jerarquia visual |
| Bounce rate | Alerta de desconexion entre origen y destino |
| Tiempo en pagina por seccion | Identifica donde se atasca o se pierde el usuario |
| Tasa de autogestion (area privada) | Mide si la web resuelve sin derivar a call center |

## 8. Principios de tono y diseno

P08. **La web habla con transparencia.** Movistar web no esconde condiciones, no maquilla precios, no entierra la letra pequena. Si una condicion es importante para la decision del cliente, debe estar visible donde se toma esa decision, no tres scrolls mas abajo.

Tono prohibido: asteriscos sin resolver, disclaimers que contradicen la oferta del hero, condiciones que solo aparecen en el ultimo paso del checkout, precios "desde" sin contexto real.

Tono correcto: precio real visible, condiciones junto a la oferta, transparencia como argumento de venta.

P09. **Cada scroll debe aportar valor.** Si un bloque no mueve al usuario un paso mas cerca de la decision, sobra. El scroll infinito sin jerarquia es el equivalente web de la tienda saturada de soportes.

P10. **La landing no es el canal de origen.** El email activa. El banner genera interes. La landing profundiza y cierra. No debe replicar el mensaje del canal de origen: debe continuarlo y resolverlo. Si el hero de la landing es identico al banner que trajo al usuario, la pagina no aporta nada nuevo.

P11. **Ventaja Personal en web debe sentirse como area VIP, no como banner mas.** En area privada, Ventaja Personal debe personalizar la experiencia: catalogo adaptado, ofertas relevantes al perfil, tono de reconocimiento. En landing publica, Ventaja Personal puede usarse como gancho ("Tienes una ventaja esperandote"), pero debe resolverse al logarse, no con un descuento generico.

## 9. Reglas de validacion automatica

| ID | Regla | Tipo |
|---|---|---|
| W01 | La pagina no tiene mision unica identificable | RECHAZO |
| W02 | El hero no comunica valor claro en los primeros tres segundos | RECHAZO |
| W03 | Hay mas de un CTA principal visible en el mismo viewport | CORRECCION |
| W04 | El precio requiere asteriscos o letra pequena para entenderse | CORRECCION |
| W05 | Las condiciones comerciales no estan visibles junto a la oferta | CORRECCION |
| W06 | Hay saturacion de ofertas o banners compitiendo en la misma pagina | RECHAZO |
| W07 | Ventaja Personal aparece como banner generico en area privada | CORRECCION |
| W08 | La landing replica el mensaje del canal de origen sin aportar profundidad | CORRECCION |
| W09 | La experiencia mobile es una adaptacion de desktop, no un diseno propio | ALERTA |
| W10 | Las FAQs contienen marketing disfrazado en vez de respuestas reales | CORRECCION |
| W11 | El CTA usa texto ambiguo ("Saber mas") en vez de accion concreta ("Contratar") | CORRECCION |
| W12 | La pagina mezcla misiones (conversion + gestion + fidelizacion en la misma landing) | RECHAZO |

Tipos: RECHAZO = la pieza no puede aprobarse. CORRECCION = debe modificarse antes de aprobacion. ALERTA = requiere justificacion editorial explicita.

## 10. Referentes de calibracion

| Referente | Atributo principal aplicable |
|---|---|
| Apple.com | Hero con un solo mensaje. Precio claro. Flujo de compra sin friccion. |
| Revolut | Transparencia radical en condiciones. Landing de producto = una idea, un CTA. |
| Stripe | Claridad tecnica traducida a beneficio humano. Documentacion como experiencia. |
| N26 | Area privada como experiencia premium. Personalizacion sin presion comercial. |

---

## 11. Checklist rapido para agentes

Antes de validar cualquier pieza web:

- [ ] El precio se entiende sin asteriscos ni letra pequena.
- [ ] El primer viewport comunica valor claro.
- [ ] Hay un unico CTA principal visible por pantalla.
- [ ] La pagina tiene una sola mision definida.
- [ ] Ventaja Personal se siente personalizada, no generica.
- [ ] Hay coherencia de mensaje entre el canal de origen y la landing.
- [ ] La experiencia mobile esta disenada, no adaptada.
- [ ] Las condiciones estan visibles, no escondidas.
- [ ] No hay saturacion de ofertas ni banners compitiendo.
- [ ] Las FAQs responden objeciones reales del cliente.
- [ ] La landing aporta profundidad, no repite el canal de origen.
- [ ] La pieza cumple las reglas de validacion automatica (W01-W12).

---

## Como usan los agentes este playbook

- **Planner (Estrategia):** lo carga si web esta en `rol_canales` del brief. Define si web es destino de trafico de otros canales (caso habitual) o canal con trafico organico propio. Asigna la mision de cada pagina.
- **Creative Copywriter (Builder):** lo carga para escribir la estructura de copy de la landing por secciones. Respeta la regla de un CTA y un foco. No repite el copy de la pieza de origen.
- **Art Director (Design):** genera mockups de landing (responsive desktop + mobile) con secciones y jerarquia clara. El mockup es conceptual; la version final pasa por el design system de movistar.es.
